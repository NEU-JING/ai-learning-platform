## 1. 背景与问题陈述

### 1.1 现状分析

AILP V3 的定位是「AI 导师全程陪伴的多路径实战学习平台」，覆盖 6 阶段课程（Python → Math → ML → DL → LLM → Engineering）+ 沙箱代码执行 + AI Tutor。

**但存在根本性定位矛盾**:

```
V3 的自我认知：                    V3 实际在做的事：
─────────────────────────────────────────────────────────────
"AI 导师陪伴的学习平台"            60% 课程内容
"Learn Your Way" 个性化           20% 交互设计
                                  10% 商业定价
                                  10% 验证/认证（后加，主体性弱）
```

### 1.2 核心问题

| 问题 | 现象 | 根因 |
|------|------|------|
| **定位模糊** | 用户不知道 AILP 与 Coursera/极客时间的区别 | 产品定义自己是「学习平台」，陷入同质化竞争 |
| **价值链条断裂** | 用户学完无法证明能力，雇主无法验证 | 缺少「验证」这一环的完整设计 |
| **实训与验证脱节** | 沙箱能跑代码，但无法生成可信的能力证据 | 缺乏四级认证体系和信任基础设施 |
| **算力瓶颈** | 2 核 4G 服务器无法支撑 Phase 3-6 完整实训 | 没有设计「混合执行模型」来分离教学与算力 |

### 1.3 重构机会

AI 人才价值链存在结构性空白：

```
教知识        练能力        验证能力        找工作
────────────────────────────────────────────────────────→
Coursera      DataCamp      ???            Boss直聘
DeepLearning.AI 七月在线    LeetCode(算法)   猎聘
极客时间      慕课网        (AI领域空白)     LinkedIn

做得好：内容   做得好：互动   做得好：没有人    做得好：流量
做不好：动手   做不好：信任   做不好：没人做    做不好：筛选
```

**空缺就在「验证 AI 能力」这一环。难度本身就是壁垒。**

## 2. 产品目标

### 2.1 一句话定义

> **AILP V4 是 AI 能力的可信验证平台——以实训为入口，以验证为价值，以招聘为自然结果。**

### 2.2 核心价值主张

| 维度 | V3（旧） | V4（新） |
|------|---------|---------|
| **我是谁** | AI 学习平台 | AI 能力验证平台 |
| **用户为什么来** | 想学 AI | 想证明自己会 AI（学是手段） |
| **用户为什么留下** | 内容好、AI Tutor 有用 | 验证结果被雇主认可 |
| **用户为什么付费** | 解锁更多内容 | 获得可分享的认证 + 雇主可见 |
| **护城河** | 课程质量 + AI Tutor | 验证数据的网络效应 + 课程演进机制 |
| **竞争对手** | Coursera、DataCamp | 没有直接对手（验证一环几乎空白） |

### 2.3 三条设计铁律

1. **Code First, Text Second** —— 任何概念讲解后，必须在同一页面内让用户写代码验证
2. **Evidence Over Claims** —— 系统只展示「用户在沙箱里做了什么」，不展示「用户说自己会什么」
3. **Verify, Don't Teach** —— 终极形态不是教得更好，而是验证得更准

## 3. 用户画像

### 3.1 主要用户群体

| 用户 | 目标 | 痛点 | AILP 价值 |
|------|------|------|----------|
| **AI 学习者** | 系统学习 AI，获得能力认证 | 学完无法证明，简历上写「熟悉 PyTorch」没人信 | 四级认证 + 公开能力主页，用数据说话 |
| **转行者** | 从传统开发转 AI 方向 | 没 AI 项目经验，面试被问住 | 完整实训记录 + 可验证的项目作品 |
| **雇主/HR** | 筛选有真实 AI 能力的候选人 | 简历造假难识别，技术面试成本高 | 扫码验证 → 看完整学习数据和代码证据 |
| **招聘平台** | 提升匹配质量 | 缺少可信的 AI 能力数据维度 | API 接入 AILP 验证数据，赋能筛选 |

### 3.2 用户使用场景

**场景 1：学习者证明能力**
```
小王完成 AILP Phase 3-6，获得 L3 Senior AI Engineer 认证
→ 生成公开主页 ailp.com/p/wangxiao
→ 简历上写「AILP L3 认证（扫码验证）」
→ 面试官扫码看到：技能雷达、23 个实验记录、2 个 Capstone 项目
```

**场景 2：雇主验证候选人**
```
HR 收到简历，看到「AILP L3 认证」
→ 扫码或调用 API 验证
→ 返回审计报告：代码提交记录、沙箱执行日志、AI 导师对话
→ 不是一张图，是一条完整数据链
```

**场景 3：招聘平台集成**
```
Boss 直聘接入 AILP API
→ 候选人授权后，能力数据自动同步
→ 岗位匹配时，AILP 数据作为筛选维度
→ 企业付费查看详细验证报告
```

## 5. 非目标（Out of Scope）

| 功能 | 原因 | 替代方案 |
|------|------|---------|
| **自建 GPU 集群** | 不是核心竞争力，成本不可控 | 学员用 Colab/Kaggle/AutoDL |
| **提供免费 LLM API** | 成本不可控，且不是产品价值 | 学员自备 API Key 或本地 Ollama |
| **K8s 实训环境** | 超出当前阶段 | Phase 6 用配置验证替代实际部署 |
| **直接做招聘平台** | 与 Boss 直聘等正面竞争 → 死 | 做能力数据输出接口，赋能招聘平台 |
| **承诺「找到工作」** | 做不到就失去信任 | 只承诺「能力被证明」 |
