## 为什么需要这次重构

V3 PRD 有 1291 行，覆盖了课程、漏斗、UI、AI Tutor、埋点、定价，看起来很完整。但它有一个根本性的定位矛盾：

```
V3 的自我认知：                          V3 实际在做的事：

"AI导师陪伴的学习平台"                   60% 课程内容
"Learn Your Way" 个性化                  20% 交互设计
                                         10% 商业定价
                                         10% 验证/认证（后加，主体性弱）
```

**问题**：当产品定义自己是"学习平台"，那它永远在和 Coursera、DataCamp、极客时间竞争。"我们有 AI Tutor"不是护城河——他们明天也可以加。

这次重构回答一个根本问题：**AILP 的终极形态不是一个更好的 Coursera。它应该是 AI 时代的「芝麻信用」——一个双方都信任的第三方能力验证标准。**

## 零、产品哲学——五条信条与设计铁律

### 0.1 五条信条（不可协商）

这些不是"产品原则"——是产品存在的理由。违反任何一条，AILP 就不再是 AILP。

```
信条 1: 不教选择题，只教写代码。
        → 实训的底线。没有代码执行的"学习"不是 AILP 的学习。

信条 2: 不靠简历夸自己，靠沙箱跑出来。
        → 验证的底线。所有能力断言必须有可追溯的代码执行记录。

信条 3: 不承诺找到工作，只承诺能力被证明。
        → 招聘的底线。AILP 是裁判，不是经纪人。裁判不能承诺比赛结果。

信条 4: 不做雇主的朋友，做双方的裁判。
        → 信任的底线。一旦偏向任何一方，验证就失去了公信力。

信条 5: 证书不是纸，是数据链。
        → 价值的底线。每一份认证背后是完整的实验记录、代码提交、AI 对话。
```

### 0.2 四条设计铁律

```
铁律 1: Code First, Text Second
        任何概念讲解后，必须在同一页面内让用户写代码验证。
        读十行文字不如写一行代码。

铁律 2: Evidence Over Claims
        系统不展示"用户说自己会什么"，只展示"用户在沙箱里做了什么"。
        技能雷达的每一条都有实验数据支撑。

铁律 3: Evolution Over Perfection
        课程不是一次性产品。课程演进机制才是产品。
        今天的内容不如明天的内容准——这个机制本身是壁垒。

铁律 4: Verify, Don't Teach
        终极形态不是教得更好，而是验证得更准。
        实训是手段，验证是目的。实训喂养验证，验证赋予实训意义。
```

### 0.3 产品基因对比

| 维度 | V3（旧定位） | V4（新定位） |
|------|:-----------:|:-----------:|
| **我是谁** | AI 学习平台 | AI 能力验证平台 |
| **用户为什么来** | 想学 AI | 想证明自己会 AI（学是手段） |
| **用户为什么留下** | 内容好、AI Tutor 有用 | 验证结果被雇主认可 |
| **用户为什么付费** | 解锁更多内容 | 获得可分享的认证 + 雇主可见 |
| **护城河是什么** | 课程质量 + AI Tutor | 验证数据的网络效应 + 课程演进机制 |
| **竞争对手** | Coursera、DataCamp、极客时间 | 没有直接对手（验证一环几乎空白） |
| **终极形态** | 最好的 AI 学习平台 | AI 能力的「芝麻信用」 |

## 一、价值链定位——为什么是"验证"这一环

### 1.1 AI 人才价值链地图

```
教知识              练能力              验证能力             找工作
──────────────────────────────────────────────────────────────→

Coursera            DataCamp              ????                Boss直聘
DeepLearning.AI     七月在线              LeetCode(算法)       猎聘
极客时间            慕课网               (AI领域空白)          LinkedIn
Udacity             贪心科技                                   拉勾

做得好：品牌/内容     做得好：互动/沙箱      做得好：没有人         做得好：流量/匹配
做不好：动手能力      做不好：信任验证        做不好：没人做          做不好：筛选质量
```

**空缺就在「验证 AI 能力」这一环。** 不是没有需求——是没人做得好。

### 1.2 为什么验证 AI 能力如此困难

| 算法能力验证（LeetCode 模式） | AI 能力验证（AILP 要做的事） |
|:---|:---|
| 有标准输入/输出，可自动判题 | 没有标准答案——模型设计、数据处理、系统架构都无唯一解 |
| 45 分钟解决一道题 | 可能需要数小时甚至数天完成一个端到端项目 |
| 雇主自己也会做这些题，能判断难度 | 大多数雇主面试官自己都没系统学过 AI，无法判断候选人水平 |
| 题库相对稳定 | AI 技术栈 6 个月就可能洗牌 |

**正是因为难，所以才值得做。难度本身就是壁垒。**

### 1.3 为什么不是「纯验证平台」

纯验证 = AI 版 LeetCode。致命问题：**用户从哪里来？**

LeetCode 有全球大学计算机教育体系做上游——学生上完课来刷题。AI 教育没有成熟的上游。如果 AILP 只做验证不讲实训，面临冷启动死亡螺旋：没内容 → 没人用 → 没验证数据 → 雇主不信 → 更没人用。

**实训是用户来的理由，验证是用户留的理由。**

### 1.4 为什么不是「招聘平台」

- 招聘是低频交易（3 年换一次工作），教育是高频消费（每天都在学）
- 做招聘平台 = 跟 Boss 直聘 5000 万用户正面打 → 死
- 承诺「找到工作」→ 做不到就失去信任
- 做裁判才能被双方信任，做经纪人只会被双方质疑

**AILP 不做招聘平台，做能力数据的输出接口。** 赋能招聘平台，不和它们竞争。

## 三、信任基础设施——比 LeetCode 更可信的验证体系

### 3.1 LeetCode 为什么被信任（以及为什么它不够）

LeetCode 验证的是「能不能在 45 分钟内解一道算法题」。雇主信任它的原因是：

| 信任锚点 | LeetCode 怎么做 | 为什么有效 |
|----------|:---:|------|
| 代码执行有记录 | 在线判题系统记录每次提交 | 不是截图，不是自述 |
| 时间戳不可伪造 | 服务器时间戳 | 形成可审计的时间线 |
| 排名有可比性 | 百分位排名 | 让雇主知道"多好算好" |
| 大规模采用 | 面试官自己也用过 | 知道题目难度 → 能判断候选人水平 |
| 第三方中立 | 不帮你写答案 | 只记录结果，不下判断 |

但 LeetCode 模式对 AI 能力验证有致命缺陷：

| 维度 | LeetCode 够用吗 | AI 能力验证的需求 |
|------|:---:|------|
| 验证深度 | 题目有标准答案 | AI 任务没有唯一解——需要验证过程，而非结果 |
| 时间尺度 | 45 分钟 | 可能需要数小时到数天的项目 |
| 技能宽度 | 算法 + 数据结构（窄） | 6+ 维度：编程/数学/ML/DL/LLM/工程化 |
| 进步可视 | 不追踪（只看最终分） | 必须展示成长轨迹——能力是否在持续提升 |
| 代码之外 | 不验证 | 需要验证系统设计、业务理解、风险判断 |

### 3.2 AILP 信任锚点七维矩阵

| 验证维度 | 传统方式（不可信） | AILP 方式（可信） | 信任原理 |
|----------|-------------------|-------------------|----------|
| **代码是自己写的？** | 提交截图/简历自述 | 沙箱内真实执行记录，每次运行有完整日志 | 代码 + 运行结果 + 时间戳 = 不可伪造 |
| **真的理解了概念？** | 选择题/问卷 | 代码 + AI 导师对话记录 | AI 提问用户回答，不是选择题——需要真懂才能答对 |
| **能力有多宽？** | 自述技能列表 | 技能雷达（6+ 维度，基于 >100 次实验数据计算） | 每个分数都有对应的实验数据支撑 |
| **做过什么项目？** | 简历自述/GitHub | AILP 上完成的实验列表（含代码 + 评分 + AI 反馈） | 可追溯、可复现、可验证 |
| **能力在进步吗？** | 无法验证 | 学习热力图 + 技能趋势图（6 个月曲线） | 时间维度的持续数据，不是一次性快照 |
| **跟别人比怎么样？** | 无法比较 | 匿名排名（同岗位/同级别百分位） | 有参照系才知道"85 分"意味着什么 |
| **证书是真的吗？** | 容易伪造（PDF 截图） | 扫码验证 → 返回背后的真实学习数据 | 不是一张纸，是一条**数据链** |

### 3.3 信任等级金字塔（用户可控的可见性）

```
        Level 3: 雇主验证 API
        ┌──────────────────────────────┐
        │ POST /verify/cert/{code}     │
        │ → 返回不可伪造的学习数据       │
        │ 雇主/招聘平台可通过 API 查证   │
        │ **用户需主动授权**             │
        └──────────────────────────────┘
                    ▲
        Level 2: 公开能力主页
        ┌──────────────────────────────┐
        │ ailp.com/p/{username}        │
        │ 用户可选择公开：               │
        │ · 技能雷达（可选择隐藏维度）    │
        │ · 完成实验列表                │
        │ · 获得的认证                  │
        │ · 同级排名                   │
        │ **用户随时可关闭公开**         │
        └──────────────────────────────┘
                    ▲
        Level 1: 个人学习数据
        ┌──────────────────────────────┐
        │ Dashboard（仅自己可见）        │
        │ · 完整技能评分 + 趋势          │
        │ · 所有实验记录                │
        │ · AI 导师对话历史             │
        │ · 成就 + 徽章                 │
        └──────────────────────────────┘
```

**关键设计**：用户主动选择公开什么。AILP 不做任何未经授权的数据暴露。这让用户信任平台——他们的数据是安全的，公开是自愿的。

## 四、课程体系 2.0——从静态课程到动态演进引擎

### 4.1 核心命题转换

| V3 思维 | V4 思维 |
|---------|---------|
| "我们有一套好课程" | "我们有一套课程**持续变好**的机制" |
| 课程更新靠人工判断 | 课程更新靠 JD 信号驱动 + 人工校准 |
| 所有课程同等对待 | 区分稳定层（1-2 年不变）和动态层（3 个月更新） |
| 课程和验证是两件事 | 课程围绕能力模型设计，验证反哺课程改进 |

```
(线性代数)       (反向传播)        (MLOps)          (LangChain)

                 (RL 原理)         (部署)           (Agent框架)

└── 稳定层 ──────┘└────── 动态层 ──────────────────┘
```

| 层级 | 更新频率 | 更新方式 | 示例 |
|------|:---:|------|------|
| **稳定层** | 1-2年 | 新增案例，不替换核心 | 反向传播的数学推导不变，但加入新优化器的对比 |
| **过渡层** | 6个月 | 更新工具链示例，保持方法论 | MLOps 方法论不变，但 CI/CD 工具从 Jenkins 换 GitHub Actions |
| **动态层** | 3个月 | 完全可替换的模块化设计 | LangChain → LangGraph → CrewAI，模块热插拔 |

### 4.5 Vibe Coding 对课程体系的冲击与响应

**核心洞察**：AI 消灭的不是"需要学习的人"，而是"只学会了调包的人"。每一次编程范式跃迁都消灭了执行层、强化了决策层和验证层。


### 6.3 企业定制验证

大客户场景：某银行说「我们的 AI 工程师需要能处理非平衡数据集（欺诈检测场景）」

AILP 响应：
1. 为该银行创建定制化评估任务
2. 候选人完成评估后生成匹配报告
3. 银行 HR 直接看到候选人在欺诈检测维度上的表现

**这是 B2B 收入的核心来源**——企业不为"培训"买单（培训预算有限），但为"精准筛选"买单（招聘失误成本远高于筛选费用）。

## 九、商业模式与定价

### 9.1 谁为验证买单

V4 的核心商业逻辑：**验证降低雇主的筛选成本，提升候选人的面试通过率。**

```
  求职者：认证通过率高 → 更可能过简历关 → 愿意付费做认证 + 公开画像
  雇主：  筛选成本降低 → 招到的人更靠谱 → 愿意付费搜索 + 定制验证
  招聘平台：匹配效率提升 → 用户体验变好 → 愿意集成认证 API
  培训机构：学员就业率高 → 口碑变好 → 愿意采购实训平台
```

### 9.3 为什么不按"课程内容"收费

V3 的定价逻辑是「免费体验 → Pro 解锁全部内容」。这个模式的问题是：

1. **天花板低**：内容订阅的天花板是 ¥199/月，和 Netflix 差不多
2. **护城河弱**：内容可以被复制（录屏、截图、搬运）
3. **与定位矛盾**：如果核心价值是「验证」，那为什么收费锚点是「内容」？

V4 改为「实训免费，验证付费」：

- 学习全部免费 → 最大化用户基数 → 积累验证数据
- 认证 + 公开画像收费 → 只有想证明自己的人付费
- 雇主端收费 → B2B 收入，单价高
