## 一、产品定义

### 1.1 一句话定义

**一个让零基础到中级开发者，在浏览器里完成"学→练→测→证"全链路的AI全栈工程师培养平台。**

### 1.2 为什么做这个

市面上的AI学习资源有两个核心问题：

| 问题 | 现状 | 我们的解法 |
|------|------|-----------|
| **学了没法练** | 课程是视频/文档，看完就忘。环境搭建劝退80%的人 | 浏览器内直接写代码、跑代码，零环境配置 |
| **练了不知道对不对** | 没有自动评测，代码跑不跑得对全靠自觉 | 实验自动评分，过不过关一目了然 |

### 1.3 不是什么

- **不是MOOC**：不做视频课程，不做被动观看
- **不是LeetCode**：不做纯算法题，强调项目级实战
- **不是企业培训SaaS**：不做组织管理、LMS、SCORM对接
- **不是AI工具**：不帮用户生成代码，而是教用户自己写

## 二、目标用户

### 2.1 核心用户

**一个人，两种状态**：

```
状态A：零基础转行者
├── 背景：非CS专业或1-3年非AI开发经验
├── 痛点：想学AI但不知道从哪开始，环境搭建就劝退了
├── 目标：能独立完成一个AI项目（RAG应用/Agent/模型部署）
├── 付费意愿：个人，对价格敏感
└── 典型场景：下班后学2小时，要即时反馈，不要折腾环境

状态B：在职开发者
├── 背景：1-3年开发经验，想转AI方向
├── 痛点：会写代码但不懂AI，碎片学习无法体系化
├── 目标：掌握LangChain/Agent/MCP等框架，能做AI工程化
├── 付费意愿：个人或企业，愿意为实战内容付费
└── 典型场景：周末集中学，要项目级实战，要证书证明能力
```

**不服务的人**：AI研究员（需要论文级深度）、企业HR（需要培训管理）、K12学生（需要游戏化）

### 2.2 用户核心旅程

```
发现 → 注册 → 选课 → 学习 → 实验 → 评测 → 下一课 → 完成认证
  │                                     │
  │     关键体验：学完马上练             │
  │     关键反馈：练完马上知道对不对     │
  │                                     │
  └─────────────────────────────────────┘
          这段越短，体验越好
```

### 3.2 四大核心功能（必须做好的）

#### 核心1：课程学习

**做什么**：提供结构化的AI学习路径，从Python基础到Agent全栈开发。

| 功能 | 说明 | 优先级 |
|------|------|--------|
| 课程列表 | 按学习路径展示课程，显示难度、时长、进度 | P0 |
| 课程详情 | 课程大纲、学习目标、前置要求 | P0 |
| 章节阅读 | Markdown渲染的教程内容，支持代码高亮 | P0 |
| 章节导航 | 上下章节切换，目录跳转 | P0 |
| 学习路径 | 可视化展示推荐学习顺序和依赖关系 | P1 |

**课程体系（已有内容）**：

```
Phase 1: Python for AI (14天)
Phase 2: 数学基础 (7天)
Phase 3: 机器学习 (28天)
Phase 4: 深度学习 (14天)
Phase 5: LLM + Agent深度 (35天)  ← 扩展课程
Phase 6: AI工程化 (35天)          ← 扩展课程
Phase 7: 综合实战项目 (21天)
```

**不做什么**：
- 不做视频课程（成本高、迭代慢）
- 不做AI自动生成学习内容（质量不可控）
- 不做学习路径自定义（先做好推荐路径）

#### 核心2：在线实验

**做什么**：在浏览器内编写和运行代码，零环境配置，做完了马上知道对不对。

这是本平台最大的差异化能力。

| 功能 | 说明 | 优先级 |
|------|------|--------|
| 代码编辑器 | 浏览器内Python代码编辑，支持语法高亮 | P0 |
| 代码执行 | 沙箱内安全执行，返回结果 | P0 |
| 实验评测 | 自动评分，输出对比+单元测试 | P0 |
| 提交历史 | 查看历次提交的代码和结果 | P1 |
| 代码提示 | 提示方向但不给答案 | P2 |
| 多语言支持 | Python为主，远期支持Rust/TypeScript | P3 |

**实验的两种形态**：

```
形态A：嵌入式练习
├── 章节内嵌小练习（3-5行代码）
├── 作用：验证理解，即时反馈
├── 评测：输出对比（简单精确匹配）
└── 体验：像做填空题

形态B：独立实验项目
├── 章节关联的完整实验（50-200行代码）
├── 作用：动手实战，综合运用
├── 评测：单元测试+功能测试
└── 体验：像做课程作业
```

**沙箱安全约束**：

| 约束 | 值 | 原因 |
|------|---|------|
| 执行超时 | 30秒 | 防止死循环 |
| 内存限制 | 256MB | 防止OOM |
| 网络访问 | 禁止 | 防止外联 |
| 文件系统 | 只读 | 防止篡改 |
| 危险代码 | 拦截 | 禁止subprocess/eval/os.system等 |

**不做什么**：
- 不做协作编程（不需要多人实时编辑）
- 不做IDE级功能（不需要Git集成/断点调试）
- 不做自由沙箱（只能做课程规定的实验，不能随意运行）

#### 核心3：学习追踪

**做什么**：让学习者清楚知道自己学到哪了、学得怎样。

| 功能 | 说明 | 优先级 |
|------|------|--------|
| 章节进度 | 标记已读/进行中/已完成 | P0 |
| 实验完成状态 | 已通过/未通过/未提交 | P0 |
| 课程总体进度 | X/Y章已完成，百分比 | P0 |
| 学习统计 | 学习天数、连续打卡、累计时长 | P1 |
| 进度持久化 | 刷新页面不丢失，跨设备同步 | P0 |

**不做什么**：
- 不做社交化排行榜（不制造焦虑）
- 不做AI学习分析（MVP阶段不需要）
- 不做学习计划定制（先做好默认路径）

#### 核心4：能力认证

**做什么**：完成课程后获得可验证的能力证明。

| 功能 | 说明 | 优先级 |
|------|------|--------|
| 证书生成 | 完成课程全部章节+实验后可申请 | P1 |
| 证书下载 | PDF格式，含用户名/课程/日期/验证码 | P1 |
| 证书验证 | 通过验证码在线核验真伪 | P2 |

**不做什么**：
- 不做区块链证书（过度工程）
- 不做技能图谱（MVP不需要）
- 不做企业认可的资质认证（需要外部机构合作，现阶段做不到）

### 3.3 辅助功能（提升体验但不阻塞核心流程）

## 四、产品架构

### 4.1 技术选型

| 层 | 技术 | 选型理由 |
|----|------|---------|
| 后端框架 | FastAPI | 异步、自动文档、Python生态 |
| 数据库 | PostgreSQL | 事务型数据、结构化 |
| 缓存 | Redis | 课程缓存、会话、限流 |
| 前端 | Vanilla HTML/JS | MVP阶段够用，无构建步骤 |
| 代码编辑 | Monaco Editor | VS Code同款，体验好 |
| 代码沙箱 | Docker容器 | 安全隔离、资源可控 |
| 部署 | Docker Compose + Nginx | 简单、可水平扩展 |

### 4.2 核心数据模型

```
User ──1:N── LearningProgress ──N:1── Chapter
  │                                      │
  │                                      N:1
  │                                      │
  │                                   Course
  │                                      │
  └──1:N── LabSubmission ──N:1── Lab ──N:1── Chapter
```

**5个核心实体**：

| 实体 | 核心字段 | 说明 |
|------|---------|------|
| User | email, username, password_hash, role | 用户 |
| Course | title, level, order_index, is_published | 课程 |
| Chapter | course_id, title, content(markdown), order_index | 章节 |
| Lab | chapter_id, title, starter_code, test_cases, solution_code | 实验 |
| LabSubmission | user_id, lab_id, code, output, status, score | 提交记录 |

**2个辅助实体**：

| 实体 | 核心字段 | 说明 |
|------|---------|------|
| LearningProgress | user_id, chapter_id, status, completed_at | 学习进度 |
| Discussion/Comment | user_id, course_id, title, content | 讨论区 |

## 五、用户体验标准

### 5.1 性能标准

| 指标 | 标准 | 底线 |
|------|------|------|
| 页面首次加载 | <2s | <5s |
| API响应（非代码执行） | <200ms | <500ms |
| 代码执行返回 | <5s（含沙箱启动） | <10s |
| 课程列表（缓存命中） | <50ms | <200ms |

### 5.2 可用性标准

| 场景 | 标准 |
|------|------|
| 新用户从注册到开始学习 | <2分钟 |
| 从读完章节到开始实验 | <3次点击 |
| 从提交代码到看到评测结果 | <5秒 |
| 重新打开平台恢复上次进度 | 0次操作（自动恢复） |

### 5.3 安全标准

| 标准 | 要求 |
|------|------|
| 认证 | JWT，token 15分钟过期 |
| 密码 | bcrypt加密存储，强度校验（字母+数字，≥8位） |
| 沙箱 | 容器隔离，禁止危险操作，超时自动终止 |
| 密钥 | 生产环境禁止硬编码，必须环境变量注入 |
| CORS | 生产环境禁止`*`，配置具体域名 |
| 代码安全 | 输入校验(Pydantic)、参数化查询、XSS防护 |

## 六、产品阶段规划

### 6.1 MVP（最小可行产品）

**目标**：一个新用户能完成"注册→选课→学习→做实验→看进度"的完整流程。

| 功能 | MVP必须 | MVP后 |
|------|---------|-------|
| 用户注册/登录 | ✅ | — |
| 课程浏览+章节阅读 | ✅ | — |
| 在线代码编辑+执行 | ✅ | — |
| 实验自动评测 | ✅ | — |
| 学习进度追踪 | ✅ | — |
| 讨论区 | ❌ | P1 |
| 证书 | ❌ | P2 |
| 管理后台 | ❌ | P3 |
| 移动端适配 | ❌ | P2 |
| 智能推荐 | ❌ | P3 |

**MVP验收标准**：一个零基础用户，能在30分钟内注册、完成第一章学习、完成第一个实验、看到自己的进度。

### 6.2 迭代路线

```
MVP (当前→4周)
├── 安全修复（生产不可有硬编码密钥/CORS*）
├── 实验评测系统（能自动打分）
├── 进度持久化（刷新不丢进度）
├── 测试基础设施（覆盖率从10%→40%）
└── 基础前端优化（Toast/Loading/错误重试）

V1.0 (4-8周)
├── API完善（分页、响应模型、文档同步）
├── 前端体验升级（面包屑、移动端、导航优化）
├── Redis缓存（课程数据走缓存）
├── Alembic迁移（数据库变更可追溯）
├── CI/CD（自动化测试+部署）
└── 讨论区完善（已有代码，需打磨体验）

V2.0 (8-16周)
├── 证书系统（PDF生成+下载）
├── 学习统计仪表板
├── Docker多阶段构建优化
├── 测试覆盖率70%+
└── 管理后台MVP

V3.0 (16周+)
├── 智能推荐
├── 多语言实验环境（Rust/TypeScript）
├── 企业版功能（团队管理、批量导入）
└── 测试覆盖率80%+
```

## 七、成功指标

**完成首个实验的用户比例**（注册→完成首次实验提交）

| 留存 | 7日留存率 | >40% | 注册7天后仍活跃 |
| 价值 | 课程完成率 | >15% | 开始学习→完成课程 |
| 传播 | 推荐率 | >10% | 用户主动推荐 |

| 指标 | 目标 |
|------|------|
| API可用率 | >99.9% |
| P95响应时间 | <500ms |
| 代码测试覆盖率 | ≥80%（终态） |

## 八、边界与约束

### 8.1 做什么

- ✅ 浏览器内代码编辑和执行（Python为主）
- ✅ 结构化学习路径和进度追踪
- ✅ 自动化实验评测和评分
- ✅ 学习完成后的能力认证
- ✅ 学习社区（讨论+问答）

### 8.2 不做什么

- ❌ 视频课程（成本和迭代速度原因）
- ❌ AI生成内容（质量不可控）
- ❌ 实时协作编程（场景不强）
- ❌ 企业LMS功能（SCORM/SSO/LTI）
- ❌ 代码托管和版本管理
- ❌ 招聘/求职对接

### 8.3 技术约束

| 约束 | 原因 |
|------|------|
| 单机部署，不强制K8s | 团队规模小，Docker Compose够用 |
| 前端Vanilla JS，暂不引入React | MVP阶段减少构建复杂度 |
| 沙箱只支持Python | Rust/TS沙箱是远期目标 |
| 不用微服务架构 | 业务复杂度不需要 |
